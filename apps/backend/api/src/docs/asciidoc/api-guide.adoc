== 개요

Schemafy API

=== 에러 응답 형식

에러 발생 시 API는 RFC 9457 Problem Details 형식으로 응답합니다.

[source,json]
----
{
  "type": "about:blank",
  "title": "Bad Request",
  "status": 400,
  "detail": "Invalid request input",
  "instance": "/api/v1.0/...",
  "reason": "COMMON_INVALID_PARAMETER"
}
----

응답 필드

[cols="1,2,1"]
|===
|필드명 |설명 |타입

|`type`
|문제 유형 URI
|String

|`title`
|HTTP 상태 제목
|String

|`status`
|HTTP 상태 코드
|Number

|`detail`
|문제 상세 설명
|String

|`instance`
|문제가 발생한 요청 경로
|String

|`reason`
|서비스 내부 에러 식별자
|String
|===

== 인증

=== JWT 인증

인증이 필요한 API(`/api/...`)는 요청 시 `Authorization` 헤더에 JWT 토큰을 포함해야 합니다.

[source]
----
Authorization: Bearer {accessToken}
----

공개 API(`/public/api/...`)는 인증 없이 접근 가능합니다.

---

=== HMAC 서명

백엔드로 요청할 때 HMAC 서명을 통해 요청의 무결성을 검증합니다.

==== 필수 헤더

[cols="1,2"]
|===
|헤더 |설명

|`X-Hmac-Signature`
|HmacSHA256으로 계산된 서명 값 (hex-encoded)

|`X-Hmac-Timestamp`
|요청 시점의 Unix timestamp (밀리초)

|`X-Hmac-Nonce`
|요청별 고유 식별자 (replay attack 방지)
|===

==== 서명 생성 방법

서명은 아래의 Canonical String을 HMAC-SHA256 알고리즘으로 해싱하여 생성합니다.

[source]
----
{METHOD}\n{PATH}?{QUERY}\n{TIMESTAMP}\n{NONCE}\n{BODY_HASH}\n
----

[cols="1,3"]
|===
|항목 |설명

|`METHOD`
|HTTP 메서드 (GET, POST, PUT, PATCH, DELETE 등)

|`PATH`
|요청 경로 (예: `/api/v1.0/workspaces`)

|`QUERY`
|쿼리 스트링 (없으면 빈 문자열)

|`TIMESTAMP`
|`X-Hmac-Timestamp` 헤더 값

|`NONCE`
|`X-Hmac-Nonce` 헤더 값

|`BODY_HASH`
|요청 본문의 SHA-256 해시 (hex-encoded). 본문이 없으면 빈 문자열의 SHA-256 해시 사용
|===

==== 검증 제외 경로

다음 경로는 HMAC 검증 대상에서 제외됩니다.

- `/public/api/...` (공개 API)
- `/ws/...` (WebSocket 연결)
- `OPTIONS` 요청 (CORS preflight)

==== 에러 코드

[cols="1,1,2"]
|===
|코드 |HTTP 상태 |설명

|`HMAC_SIGNATURE_MISSING`
|401
|HMAC 서명이 누락되었습니다.

|`HMAC_SIGNATURE_INVALID`
|401
|HMAC 서명이 유효하지 않습니다.

|`HMAC_TIMESTAMP_EXPIRED`
|401
|HMAC 타임스탬프가 만료되었습니다. (허용 범위: ±30초)

|`HMAC_NONCE_DUPLICATE`
|401
|중복된 HMAC 논스입니다.
|===

==== 키 로테이션

서비스 운영 중 HMAC 비밀 키를 교체할 때, 이전 키(`previousSecret`)를 함께 설정하면 교체 기간 동안 두 키 모두 유효합니다. 새로운 키로 검증 실패 시 이전 키로 재검증을 시도합니다.

== ERD API

ERD 관련 API입니다.

=== 실시간 협업 연동

ERD 관련 mutation API(생성, 수정, 삭제)를 호출하면, 같은 프로젝트에 접속 중인 다른 사용자에게 `ERD_MUTATED` 이벤트가 WebSocket을 통해 자동으로 브로드캐스트됩니다.

==== X-Session-Id 헤더

ERD mutation API를 호출할 때 `X-Session-Id` 헤더를 함께 전송하면, 해당 세션으로 접속한 클라이언트에게는 이벤트가 전송되지 않습니다. 이를 통해 자기 자신의 변경에 대해 중복으로 UI를 갱신하는 것을 방지할 수 있습니다.

[cols="1,2,1"]
|===
|헤더 |설명 |필수

|`X-Session-Id`
|WebSocket 연결 시 할당된 클라이언트 세션 ID
|선택 (협업 환경에서 권장)
|===

NOTE: `X-Session-Id`를 전송하지 않으면 모든 접속자에게 이벤트가 브로드캐스트됩니다.

==== X-Client-Op-Id 헤더

ERD mutation API를 호출할 때 `X-Client-Op-Id` 헤더를 함께 전송하면, 서버는 해당 값을 커밋된 operation metadata에 기록합니다. 이 값은 HTTP mutation 응답의 `operation.clientOperationId`와 `ERD_MUTATED.operation.clientOperationId`로 다시 전달됩니다.

[cols="1,2,1"]
|===
|헤더 |설명 |필수

|`X-Client-Op-Id`
|클라이언트가 생성한 operation 상관관계 ID
|선택
|===

TIP: 클라이언트는 재시도 중인 동일 요청에 같은 `X-Client-Op-Id`를 재사용하여 HTTP 응답과 WebSocket 이벤트를 안전하게 매칭할 수 있습니다.

==== X-Base-Schema-Revision 헤더

ERD mutation API를 호출할 때 `X-Base-Schema-Revision` 헤더를 함께 전송하면, 서버는 요청 시점에 클라이언트가 알고 있던 schema revision을 함께 수신합니다.

[cols="1,2,1"]
|===
|헤더 |설명 |필수

|`X-Base-Schema-Revision`
|클라이언트가 요청 생성 시점에 알고 있던 schema revision
|선택
|===

NOTE: mutation 성공 후 실제 커밋 revision은 HTTP mutation 응답의 `operation.committedRevision`과 `ERD_MUTATED.operation.committedRevision`으로 확인할 수 있습니다.

==== 대상 API

다음 API들이 mutation 수행 후 `ERD_MUTATED` 이벤트를 발생시킵니다.

[cols="1,2"]
|===
|카테고리 |API

|Schema
|생성, 이름 변경, 삭제

|Table
|생성, 이름 변경, 메타 변경, 추가정보 변경, 삭제

|Column
|생성, 이름 변경, 타입 변경, 메타 변경, 위치 변경, 삭제

|Constraint
|생성, 이름 변경, CHECK/DEFAULT 표현식 변경, 삭제, 컬럼 추가/제거/위치 변경

|Index
|생성, 이름 변경, 타입 변경, 삭제, 컬럼 추가/제거/위치 변경/정렬 방향 변경

|Relationship
|생성, 이름 변경, 종류 변경, 카디널리티 변경, 추가정보 변경, 삭제, 컬럼 추가/제거/위치 변경
|===

NOTE: 조회(GET) API는 이벤트를 발생시키지 않습니다.

== WebSocket API

실시간 기능을 위한 WebSocket API입니다.

=== 프로젝트 협업

프로젝트 내 실시간 협업 기능을 제공합니다.

==== 연결

===== 엔드포인트

[source]
----
ws://{host}/ws/collaboration?projectId={projectId}
----

쿼리 파라미터

[cols="1,2,1"]
|===
|파라미터 |설명 |필수

|`projectId`
|협업할 프로젝트 ID (ULID)
|필수
|===

===== 인증

WebSocket 연결 시 다음 두 가지 방법 중 하나로 JWT 토큰을 전달해야 합니다.

방법 1: HTTP 헤더

[source]
----
Authorization: Bearer {accessToken}
----

방법 2: 쿠키

HTTP 헤더에 토큰을 포함할 수 없는 환경(브라우저 등)에서는 쿠키를 사용합니다.

[cols="1,2"]
|===
|쿠키 이름 |설명

|`accessToken`
|JWT 액세스 토큰
|===

연결이 완료되면 서버는 현재 연결된 클라이언트에게 `SESSION_READY` 이벤트를 1회 전송합니다. 이후 브로드캐스트되는 협업 이벤트에도 발신자의 `sessionId`가 포함될 수 있지만, 현재 연결의 WebSocket `sessionId`를 최초로 확인할 때는 이 이벤트를 사용하면 됩니다.

  "operation": {
