## 핵심 발견

### ✅ 렌더 횟수 — 규모와 무관하게 완전히 고정

작업별 렌더 횟수가 테이블 수에 전혀 영향받지 않는다:

- addTable: 4→6→11 테이블 모두 **8회 고정**
- addColumn: 4→6→11 테이블 모두 **4회 고정**
- connectEdge: 4→6→11 테이블 모두 **24회 고정**

React 렌더링 경계가 올바르게 분리되어 있다는 증거다.

### ✅ 반복 작업 — 열화 없음

컬럼 5회 연속 추가에서 렌더 3회, 소요시간 ~320ms가 전혀 흔들리지 않는다. 누적 상태 오염이나 메모리 누수 없음.

### ⚠️ 스타일 재계산 — 급격한 비선형 증가

`recalcStyleCount` 누적값이 241 → 366 → 632로 증가한다:

- 4→6 테이블(+2): +125회
- 6→11 테이블(+5): +266회

테이블 5개 추가에 스타일 재계산이 +266회로, 2개 추가 시 +125회의 **2.1배**다. 노드가 많아질수록 단일 렌더의 스타일 재계산 비용이 선형 이상으로 증가하고 있다.

### ⚠️ 스크립트 실행 시간 — 2배 증가

`scriptDuration` 누적값이 0.538 → 0.751 → 1.083s로 증가한다. 4 테이블 대비 11 테이블에서 **2.01배**다. 렌더 횟수는 동일한데 스크립트 실행이 늘어나는 것은, 각 렌더 사이클에서 처리하는 노드 수가 많아졌기 때문이다.

### ⚠️ 이벤트 리스너 — 불균형 급증

- 4 테이블: 412개 (~103/테이블)
- 6 테이블: 898개 (+486, connectEdge 연결 후 엣지 리스너 추가 추정)
- 11 테이블: 981개 (+83)

엣지 연결 시 엣지 자체의 이벤트 리스너가 대량 등록되는 것으로 추정된다. 엣지 1개당 수백 개의 리스너가 붙을 수 있다.

### ⚠️ 힙 total 급증 (11 테이블)

`totalMB`가 54.1 → 55.9 → **75.9MB**로 11 테이블에서 20MB 급증한다. GC가 아직 실행되지 않은 상태이거나, 이전 노드/엣지 데이터가 해제되지 않고 있을 가능성이 있다.

## 원인 분석

### 1. 스타일 재계산 가속 — React Flow 전체 노드 재계산

React Flow는 내부적으로 노드 위치 계산(ELK/내부 레이아웃)을 할 때 **모든 노드를 순회**한다. 렌더 횟수는 고정이지만 각 렌더에서 처리하는 노드 수가 증가하면 CSS 스타일 재계산이 노드 수에 비례해 증가한다. `recalcStyleCount`가 렌더 횟수와 독립적으로 증가하는 이유다.

### 2. 이벤트 리스너 급증 — 엣지 연결 시 대량 등록

4→6 테이블 전환에서 리스너가 +486 급증했는데, 이 시점에 `connectEdge`가 성공하면서 엣지가 생성됐을 가능성이 높다. React Flow의 엣지는 SVG 경로 + 오버레이 + 인터랙션 핸들러 조합으로 구성되어 개당 수십 개의 리스너가 붙는다.

### 3. 힙 total 급증 — 미회수 노드 데이터

11 테이블 시점에 `totalMB` 75.9MB는 GC 압력이 높아졌음을 나타낸다. React Flow의 내부 노드 맵, 엣지 경로 캐시, 선택 상태 히스토리 등이 해제되지 않고 누적되고 있을 수 있다.

## 해결 방법

### 1. 스타일 재계산 감소 — CSS containment 적용

각 테이블 노드에 CSS `contain: layout style`을 적용하면 브라우저가 노드 내부 변경을 외부 레이아웃과 격리하여 전체 스타일 재계산을 방지한다.

```css
/* 테이블 노드 래퍼 */
.react-flow__node-table {
  contain: layout style;
}
```

### 2. 이벤트 리스너 최적화 — 엣지 수동 cleanup

엣지 제거 시 이벤트 리스너가 자동으로 정리되지 않는다면, React Flow의 `onEdgesDelete` 콜백에서 명시적 cleanup을 추가한다.

```ts
const onEdgesDelete = useCallback((edges: Edge[]) => {
  // 엣지 관련 구독/리스너 명시적 해제
  edges.forEach((edge) => cleanupEdgeListeners(edge.id));
}, []);

<ReactFlow onEdgesDelete={onEdgesDelete} ... />
```

### 3. 힙 압력 완화 — React Flow 노드 메모이제이션

노드 배열이 매 렌더마다 새로 생성되면 GC 압력이 증가한다. `useMemo`로 노드·엣지 배열을 안정화한다.

```ts
const nodes = useMemo(() => tableList.map(tableToNode), [tableList]);
const edges = useMemo(() => relationList.map(relationToEdge), [relationList]);
```

### 4. 장기 대응 — React Flow `onlyRenderVisibleElements`

스타일 재계산과 스크립트 실행 증가의 근본 원인은 뷰포트 밖 노드도 렌더에 포함되기 때문이다. 이 옵션 하나로 뷰포트 외 노드를 DOM에서 제거해 모든 지표를 개선할 수 있다.

```tsx
<ReactFlow onlyRenderVisibleElements={true} nodes={nodes} edges={edges} />
```
